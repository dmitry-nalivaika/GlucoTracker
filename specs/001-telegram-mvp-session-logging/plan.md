# Implementation Plan: GlucoTrack Telegram MVP — Session Logging & AI Analysis

**Branch**: `001-telegram-mvp-session-logging` | **Date**: 2026-04-17 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/001-telegram-mvp-session-logging/spec.md`

---

## Summary

Build a Telegram bot that lets users log meal sessions (food photos + CGM screenshots + activity notes),
triggers Claude-powered AI analysis of each session, delivers structured analysis back in Telegram,
and posts a summary card to a Miro board. All data is isolated per Telegram user ID. The data model
supports future cross-session trend analysis from day one.

---

## Technical Context

**Language/Version**: Python 3.11
**Primary Dependencies**: `python-telegram-bot` v22, `anthropic` v0.40+, `sqlalchemy` 2.0, `httpx`, `alembic`, `pydantic` v2, `ruff`, `black`, `mypy`, `pytest`, `pytest-asyncio`, `pytest-cov`
**Storage**: SQLite (via SQLAlchemy 2.0 async engine) for dev/MVP; local filesystem for files; designed for Azure SQL + Azure Blob migration
**Testing**: `pytest` + `pytest-asyncio` + `pytest-cov` + `respx` (httpx mock) + `unittest.mock`
**Target Platform**: Linux server (Azure App Service B1 or Azure Container Apps)
**Project Type**: Telegram bot service
**Performance Goals**: Telegram acknowledgement < 2s (SC-002); AI analysis < 30s (SC-003); Miro card < 5s after analysis (SC-004)
**Constraints**: $50/month hard cap (Constitution VII); 80% code coverage minimum (Constitution V); all queries user-scoped (Constitution II)
**Scale/Scope**: Up to 50 sessions/day (SC-007); ≤ 10 AI calls/user/day rate limit

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-checked after Phase 1 design.*

### Principle II — Multi-User Architecture & Data Isolation ✅

- **Query scoping**: Every repository method takes `user_id` as a required parameter. No query runs without it. Enforced at the SQLAlchemy repository layer.
- **File paths**: All user files stored at `{STORAGE_ROOT}/users/{user_id}/sessions/{session_id}/`. `StorageRepository` enforces this pattern; no direct path construction outside it.
- **ACL sharing**: Not implemented in this feature (explicitly out of scope per spec Assumptions). Architecture supports it — `user_id` FK on every entity makes sharing implementable via ACL table in a future feature.
- **Audit log**: Not required for this feature (no sharing/revocation actions).
- **Telegram identity**: Telegram user ID is the user identity for MVP. All stored artefacts tagged at write time.

**GATE: PASSED**

### Principle III — Technology Stack ✅

- Python ✓ (Python 3.11)
- Anthropic Claude API ✓ (`anthropic` SDK, isolated in `AIService`)
- Telegram Bot API ✓ (`python-telegram-bot` v22, behind `BotAdapter` interface)
- Miro Board API ✓ (`httpx` client, isolated in `MiroService`)
- Claude API isolated in dedicated AI service module ✓
- Input channels behind interface ✓ (`BotAdapter` protocol)
- Storage drivers abstracted behind repository interface ✓ (`StorageRepository` protocol)

**GATE: PASSED**

### Principle V — Code Quality & Testing ✅

- TDD: tests written first, must fail before implementation
- Coverage ≥ 80% enforced in CI
- `ruff` + `black` + `mypy` in CI pipeline
- Parameterised queries via SQLAlchemy ORM (no string concatenation)
- All user input validated at Telegram handler boundary
- No `print()` — structured logging via `logging` module
- No hardcoded secrets — all from environment variables

**GATE: PASSED**

### Principle VII — Cost Management ✅

- Per-user rate limit: 10 AI analysis calls/user/day (configurable)
- Per-session token budget: 4,000 output tokens max
- In-memory token bucket (sufficient for single-instance MVP)
- Azure App Service B1 cost estimate: ~$13/month
- Anthropic API at 50 sessions/day × 4000 tokens × $3/M tokens ≈ $18/month
- Miro: free tier (up to 3 boards, unlimited cards)
- Total estimated: ~$31/month — within $50 cap ✓

**GATE: PASSED**
